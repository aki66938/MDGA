# Plan18 - 视觉流水线：意图驱动的「主 agent ↔ 视觉模型」通信

项目代号：MDGA
文档定位：Plan17 搭好了视觉 provider 的配置槽与图像门禁，但**没有真正的识图**。本文设计让视觉**真正生效**的一套逻辑：在「文本主 agent + 独立视觉模型」两个模型之间，建立一套**意图驱动的通信交互系统**，使主 agent 能借视觉模型读懂用户上传图片的用意并据此完成需求。

> ⚠️ **本计划待主创审阅通过后再编码。** 决策处标注默认建议，可改。

---

## 1. 为什么需要专门设计（与 CC/Codex 的本质区别）

- Claude Code / Codex 的底模**本身就是多模态**：图片直接进同一个模型上下文，无需任何「模型间通信」。
- MDGA 的架构是**主 agent（文本，如 DeepSeek）+ 可插拔视觉模型（如 GLM-4V / Qwen-VL / Anthropic 格式模型）**，两者是**两个独立模型**、两条独立 API。主 agent **看不见图片**。
- 因此必须有一座**桥**：把图片交给视觉模型「看」，把视觉模型的「所见」转成**文本**喂回主 agent。难点不在「能不能看」，而在**让视觉模型看对地方**——即带着用户的**用意**去看，而不是泛泛描述。

## 2. 用意（intent）从哪来——本设计的核心

用户上传图片的「用意」有两个来源（主创已指明）：
- **（主）主 agent 从用户侧收到的需求文本**：如「把这张设计图实现成 React 组件」「这个报错怎么解决」「整理这张表格的数据」。这是视觉查询的**方向盘**。
- **（次）图片本身 + agent 与视觉模型之间的提示词**：视觉模型据图内容 + 桥接提示词补充细节。

**关键原则：视觉模型绝不做「无意图的泛化描述」。** 每次视觉调用都把用户意图作为「看什么」的指引，让视觉模型只提取**与需求相关**的信息、要点化输出。

## 3. 通信交互系统（核心流程）

```
用户：[图片] + 「把这张设计图实现成 React 组件」
        │
        ▼ 前端把图片(base64)随消息发给后端
┌─────────────────────────────────────────────┐
│ 后端 send_message：检测到图片 + 已配视觉 provider │
│                                               │
│ ① 自动「初看」一遍（intent-contextualized）：    │
│   视觉请求 = system(视觉分析助手 + 输出要求)      │
│            + user(用户意图「实现成React组件」+ 具体问题) │
│            + image(s)                          │
│   → 调视觉模型 → 得到「针对需求的要点化文本分析」    │
│                                               │
│ ② 把分析作为上下文注入主 agent：                  │
│   system: "[视觉分析] 用户上传了N张图，针对其需求， │
│            视觉模型识别如下：{analysis}。请据此继续" │
└─────────────────────────────────────────────┘
        │
        ▼ 主 agent（文本）据「文本化的图」+ 用户需求，正常跑工具循环完成任务
        │
        ▼（若信息不够）主 agent 调 ask_vision(具体问题) 工具
           → 后端把相关图 + 该问题再发给视觉模型 → 返回文本 → 回灌主 agent
```

**两段式桥接**：
- **① 自动初看（必做）**：图片一上传就带意图过一遍视觉，产出「图的文本化」注入主 agent。保证主 agent 一开局就「知道图里有什么、与需求相关的部分」。
- **② ask_vision 追问（按需）**：主 agent 在干活中若需更细节（「左上角按钮的具体颜色值是多少」「第 3 行报错完整文本」），调 `ask_vision(question)` 工具，把图 + 具体问题再发视觉模型，返回文本。实现迭代式精读。

### 桥接提示词设计（视觉模型的 system + user）
- **system**：「你是视觉分析助手。根据给定的用户需求，仔细观察图片，提取与该需求**直接相关**的所有信息（布局 / 文字 / 数据 / 颜色 / 尺寸 / 结构 / 报错内容 / 代码等），以要点化、信息密集的中文输出，供一个**看不到图片**的文本模型据此完成需求。只描述与需求相关的内容，不泛泛而谈、不寒暄。」
- **user**：`用户需求：{intent}\n{可选的具体问题}` + 图片。

## 4. 视觉 provider：OpenAI + Anthropic 双格式（主创定）

视觉模型 **base_url 用户自填**，并**同时兼容 OpenAI 与 Anthropic 两种 API 格式**，方便用户照各平台 API 文档配置。在视觉 provider 配置里新增「**API 格式**」选择：`openai` | `anthropic`。

| | OpenAI 格式 | Anthropic 格式 |
|---|---|---|
| 端点 | `{base_url}/chat/completions` | `{base_url}/v1/messages` |
| 鉴权头 | `Authorization: Bearer {key}` | `x-api-key: {key}` + `anthropic-version: 2023-06-01` |
| 图片消息 | `content:[{type:"text",text},{type:"image_url",image_url:{url:"data:image/png;base64,…"}}]` | `content:[{type:"text",text},{type:"image",source:{type:"base64",media_type:"image/png",data:"…"}}]` |
| 取回文本 | `choices[0].message.content` | `content[0].text` |

实现：视觉调用走一个**独立的 vision 客户端**（新增 `crates/vision-client` 或扩展 deepseek-client），按 provider 的 `api_format` 分支构造请求与解析响应。`max_tokens` 等必填项按格式给默认。

## 5. 数据与消息模型

- **图片附带**：前端 composer 选图后，把图片读为 base64（含 media_type）随 `send_message` 上送；会话内消息 parts 增加 `image` 类型（`{ mediaType, dataRef }`）。
- **图片存储**（见 D2）：base64 体积大，建议落盘到 app-data/会话目录，消息里存**引用路径**而非内联 base64，避免撑大 SQLite；ask_vision 时按引用重新读图。
- **token 账本**：视觉调用消耗记为**独立的 vision usage**条目（与主模型分开计费/展示），呼应「成本透明」底色。

## 6. 关键设计决策（审阅可改）

| # | 决策 | 默认建议 |
|---|---|---|
| D1 | 触发方式 | **自动初看 + ask_vision 追问** 两段式（初看保证开局认知，追问按需精读） |
| D2 | 图片存储 | 落盘 app-data/会话目录 + 消息存引用（不内联 base64 进 DB） |
| D3 | API 格式 | provider 配置里**用户显式选** openai / anthropic（默认 openai；两者端点/头/结构不同，不做自动探测） |
| D4 | 视觉分析可见性 | 在对话流内以**可折叠「视觉分析」卡片**展示（透明、可审、让用户看到识图结果与成本），默认折叠 |
| D5 | 多图 | 单次消息支持多图，一并送视觉；ask_vision 默认针对该消息全部图 |

## 7. 里程碑

- **M18.1 视觉真正生效（核心）**：vision 客户端（OpenAI + Anthropic 双格式）+ 视觉 provider 增 `api_format` 字段与配置 UI + composer 图片附带(base64)+ 图片存储 + 自动初看 pass（意图驱动）+ 分析注入主 agent。验收：配好视觉模型后，上传 UI 设计图 + 「实现成组件」→ 主 agent 能据视觉分析产出代码。
- **M18.2 精读与透明**：`ask_vision(question)` 追问工具 + 对话流「视觉分析」可折叠卡片 + 视觉 usage 独立记账 + 多图。

## 8. 不在本计划（后续）

- **订阅制模型**（coding plan / token plan 等非 key 鉴权）：主创定**暂缓**，待后续 provider 模块深入完善时再加。
- **音频 / 语音**模态（Plan17 已占位）。
- 视觉模型反向调用工具、视觉 agent 自主多轮等高级编排。

## 9. 验收（本计划范围）

- 视觉 provider 可选 OpenAI / Anthropic 格式、自填 base_url，配置后能成功调用。
- 上传图片 + 文本需求 → 自动初看产出意图相关的分析 → 主 agent 据此完成需求（如设计图→代码、报错图→修复建议、表格图→数据整理）。
- ask_vision 追问可获取图中更细节并回灌主 agent。
- 未配视觉时维持 Plan17 现状（拒图）；视觉 usage 在账本可见。
- `cargo test --workspace` + `tsc` 全绿；主创 dev 真机走查（含 OpenAI 与 Anthropic 两种格式各配一个真实视觉模型验证）。
